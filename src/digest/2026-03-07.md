### [Clinejection — Compromising Cline's Production Releases Just by Prompting an Issue Triager](https://adnanthekhan.com/posts/clinejection/)

Security researcher Adnan Khan demonstrated a devastating attack chain against the Cline GitHub repository: a prompt injection in a GitHub issue title tricked an AI-powered issue triage workflow (using `claude-code-action`) into installing a malicious npm package via `preinstall` scripts, which then poisoned the shared GitHub Actions cache — ultimately compromising the nightly release workflow and stealing NPM publish secrets. Cline's `v2.3.0` was briefly published by the attacker before being retracted.

> This is required reading for anyone building AI-powered workflows with tool access. The attack surface isn't the LLM itself — it's the combination of untrusted input, permissive tool grants, and shared infrastructure. In enterprise settings, the equivalent risks are AI agents with access to internal APIs, shared cache layers, and production pipelines. Prompt injection is not an academic concern; it's an active threat to CI/CD supply chains.

### [Agentic Manual Testing (Simon Willison — Agentic Engineering Patterns)](https://simonwillison.net/guides/agentic-engineering-patterns/agentic-manual-testing/)

Simon Willison details a pattern for having coding agents manually test their own output — using `python -c`, `curl` against dev servers, and Playwright/browser automation via tools like Rodney. The key insight: passing automated tests doesn't mean code works; agents can use visual screenshots and accessibility trees to validate UI behavior beyond what unit tests cover.

> The "show your work" principle for agents is underappreciated in enterprise AI discussions. Agents that execute and verify their own code — rather than just generating it — are fundamentally more reliable. This is directly applicable to any agentic coding workflow: building in a self-verification loop (run it, check the output, fix and re-run) should be a standard pattern, not an afterthought.

> AI-driven vulnerability research at scale is a real capability now, not a research demo. The velocity here (22 vulns in 2 weeks across a mature, well-audited codebase) should recalibrate expectations for what AI agents can accomplish on focused, bounded tasks.

### [AINews: AI Engineer Will Be the LAST Job](https://www.latent.space/p/ainews-ai-engineer-will-be-the-last)

Latent Space's latest AINews digest digs into the "AI Engineer is the last job" thesis: both OpenAI and Anthropic now claim AI can handle ~70% of white-collar work; software engineering already accounts for over 50% of Claude model usage; and 2026 is being called the "Year of Knowledge Work Agents." The Jevons Paradox argument: as AI automates software engineering tasks, demand for AI Engineers who build and orchestrate those systems actually increases — and may be the last human role standing.

> The 50%-of-Claude-usage statistic is worth sitting with. It means the primary value of today's most capable AI is being captured by the people who build with it, not just the people who use it. For enterprise architects: the competitive advantage in the next 3-5 years isn't in adopting AI tools, it's in building the AI-native workflows and orchestration layers that compound over time. The "last job" framing is provocative, but the underlying observation — that AI engineers uniquely benefit from Jevons Paradox — is solid.
