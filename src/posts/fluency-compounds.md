There are two types of AI users right now. They look the same from the outside. They're not.

The first type uses AI to speed up existing tasks. They type a prompt, get output, tweak it. They're 20-30% faster. That's real. It's also the ceiling.

The second type has stopped describing *how* to do things. They describe *outcomes*. Not "write a function that parses this JSON" but "I need the system to guarantee this invariant holds." The AI works backwards from there. They're not faster — they're operating at a different altitude entirely.

Here's what makes this dangerous: the gap compounds.

Every time a fluent user interacts with AI, they're doing architecture and outcome-definition. They're getting sharper at those things. Meanwhile, the first type is getting faster at implementation — a skill the AI is also getting better at every month, which means it's worth less every month.

Two engineers. Same tools. Same team.

- **Engineer A** uses Copilot to write code faster. Closes the same tickets, slightly quicker.
- **Engineer B** uses AI to skip implementation entirely. More time on specs, interfaces, failure modes. Fewer tickets, bigger decisions.

A year from now, Engineer B has done 12 months of architecture thinking. Engineer A has done 12 months of fast typing.

The engineering metrics won't catch this. They count commits and PRs. Neither measures what level you're operating at.

The tell is how someone talks to AI. Fluent users write specs, not prompts. They care about what "done" looks like, not what the code looks like. It's the same shift I wrote about in [The Diff Is Dead](/posts/the-diff-is-dead/) — the bottleneck moves upstream. Fluency just accelerates how far upstream it goes.

The people who figured this out early are compounding quietly. The people still treating AI as a faster keyboard are heading toward the wrong kind of efficiency.
